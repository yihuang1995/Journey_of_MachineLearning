# How to Evaluate and Launch a New Recommendation Model

This note discusses how to evaluate and launch a new version of a recommendation model, especially when the new model changes the architecture, feature set, objective function, or ranking policy.

The key idea is that recommendation model evaluation is different from standard supervised learning evaluation. In standard regression or classification problems, we often have labels for most examples. But in recommendation systems, user feedback is generated under the old recommendation policy. Users only react to items, rows, or slates that were actually shown to them. For items that were not exposed, we do not observe the counterfactual outcome.

Because of this, recommendation model evaluation usually follows a multi-stage funnel:

```text
1. Offline evaluation / backtesting
2. Counterfactual evaluation, if possible
3. Shadow testing / canary rollout
4. Online A/B testing
```

Each stage answers a different question:

```text
Offline evaluation:
Does the new model look better on historical data?

Counterfactual evaluation:
Can we estimate how the new policy would have performed using old logs?

Shadow testing:
Does the model run correctly in production without affecting users?

Canary rollout:
Does the model behave safely on a small amount of real traffic?

A/B testing:
Does the model causally improve online business metrics?
```

---

## 1. Why Recommendation Evaluation Is Different

For standard supervised learning problems, such as regression or classification, the model usually predicts an outcome that exists independently of the model decision.

Examples:

```text
Credit risk:
User defaulted or did not default.

Fraud detection:
Transaction was fraud or not fraud.

LTV prediction:
User generated certain future revenue.

Classification:
User converted or did not convert.
```

In these cases, the label is often observable after some time, even if there are still issues like label delay, selection bias, or noisy labels.

Recommendation systems are different because the model directly determines what the user sees. The user can only click, buy, skip, watch, or engage with content that was actually exposed.

For example:

```text
Old model showed item A to user U.
User clicked item A.

But the new model might have shown item B.
We do not know whether user U would have clicked item B.
```

So the observed label is conditional on the old model’s action.

A more formal way to describe this:

```text
Context: x
Action shown by old model: a_old
Observed reward: R(x, a_old)

New model wants to choose: a_new
But we usually do not observe: R(x, a_new)
```

If:

```text
a_new != a_old
```

then historical logs usually do not contain the true reward for the action the new model would have taken.

This is the core counterfactual problem.

---

## 2. Main Biases in Recommendation Logs

Historical recommendation logs are biased because they are generated by the previous system.

Common biases include:

```text
Exposure bias:
Only shown items have labels.

Position bias:
Items or rows shown higher receive more attention and more clicks.

Selection bias:
The old model chose which candidates were shown, so logged data reflects the old model’s preferences.

Popularity bias:
Popular items get more exposure, more feedback, and continue to look better.

Feedback loop:
The model affects exposure, exposure affects feedback, and feedback affects the next model.

Missing counterfactuals:
We do not observe user responses to unshown items or alternative rankings.
```

This is why offline evaluation is useful but not sufficient.

---

## 3. Stage 1 — Offline Evaluation / Backtesting

Offline evaluation is still the first step. It helps filter out bad models before any online exposure.

The goal is to answer:

```text
Does the new model improve prediction or ranking quality on historical logs?
```

Typical setup:

```text
Train data:
Historical interactions before time T.

Validation / test data:
Interactions after time T.

Compare:
Old model vs new model on the same offline data.
```

Example:

```text
Train: January to March interactions
Test: April interactions
Compare old ranking model and new ranking model on April logs
```

### Common Offline Metrics

For retrieval models:

```text
Recall@K
HitRate@K
NDCG@K
MAP@K
MRR
embedding coverage
candidate diversity
candidate freshness
latency
```

For ranking models:

```text
AUC
log loss
NDCG@K
MRR
calibration
CTR prediction quality
CVR prediction quality
ranking lift by segment
```

For e-commerce recommendation:

```text
add-to-cart prediction
purchase prediction
GMV prediction
profit prediction
conversion ranking quality
```

For feed / content recommendation:

```text
watch time prediction
long-watch prediction
like/share/comment prediction
retention-related engagement
negative feedback prediction
```

### What Offline Evaluation Is Good For

Offline evaluation is useful for:

```text
Checking whether the model learned useful patterns
Comparing model architectures
Debugging feature quality
Checking calibration
Checking segment-level performance
Filtering out clearly bad models
Reducing online experiment cost
```

### What Offline Evaluation Cannot Fully Answer

Offline evaluation cannot fully answer:

```text
Would users behave differently if exposed to new recommendations?
Would the new model change item exposure distribution?
Would long-tail items perform better if shown?
Would the model improve real online metrics?
Would the model create unexpected ecosystem effects?
```

The reason is that offline logs only contain feedback under the old policy.

So offline evaluation is a necessary but not sufficient condition for launch.

---

## 4. Stage 2 — Counterfactual Evaluation / Off-Policy Evaluation

Counterfactual evaluation, also called off-policy evaluation, tries to answer:

```text
Using logs generated by the old recommendation policy, can we estimate how the new recommendation policy would have performed?
```

This is especially useful when running a full A/B test is expensive or risky.

### Required Logged Data

For each historical request, we ideally need:

```text
x: context features
a: action shown by old policy
r: observed reward
p_old(a | x): probability that the old policy chose action a under context x
```

The log format is:

```text
(x, a, r, p_old(a | x))
```

For recommendation systems, the action may be:

```text
an item
a row/module
a ranked list
a slate
a row at a specific position
```

The reward may be:

```text
click
add-to-cart
purchase
watch time
long-watch
GMV
profit
follow
share
retention proxy
```

The most important requirement is the old policy propensity:

```text
p_old(a | x)
```

This tells us how likely the old system was to show that action.

Without propensity logging or randomized exploration, counterfactual evaluation becomes much weaker.

---

## 5. Why Exploration Is Needed

Exploration is needed because the old recommendation system usually does not expose every reasonable action.

Without exploration, we only know:

```text
Old model chose item A → user response to A
```

But we do not know:

```text
What if we had shown item B?
What if row X had been placed at position 1?
What if a long-tail item had been exposed?
```

If the old policy never shows item B under a certain context, then:

```text
p_old(B | x) = 0
```

This creates a lack of support or no-overlap problem. In that case, we cannot estimate the effect of a new policy that wants to show B.

Controlled exploration gives many actions a non-zero probability of being shown:

```text
p_old(a | x) > 0
```

This enables counterfactual evaluation.

Examples of exploration:

```text
epsilon-greedy exploration
random item swaps
random row swaps
randomized top-K ordering
Thompson sampling
UCB
small traffic randomization
```

In production, exploration should be controlled:

```text
small percentage of traffic
safe candidate set only
quality thresholds
business guardrails
latency guardrails
real-time monitoring
```

Exploration is not random chaos. It is controlled data collection.

---

## 6. IPS — Inverse Propensity Scoring

The simplest counterfactual estimator is IPS.

The goal is to estimate the value of the new policy:

```text
V(new policy)
```

The IPS estimator is:

```text
V_IPS = (1 / N) * sum_i [pi_new(a_i | x_i) / pi_old(a_i | x_i)] * r_i
```

Where:

```text
x_i = context
a_i = action shown by old policy
r_i = observed reward
pi_old(a_i | x_i) = old policy probability
pi_new(a_i | x_i) = new policy probability
```

The importance weight is:

```text
w_i = pi_new(a_i | x_i) / pi_old(a_i | x_i)
```

### Intuition

If the old model rarely showed an action, but the new model would show it more often, then observations of that action should receive higher weight.

Example:

```text
Old policy showed item A with probability 0.1.
User clicked item A.
Reward = 1.

If the new policy would show A more often, this sample is important.
Weight = 1 / 0.1 = 10.
```

This means a rare but informative exposure is upweighted.

### Simple IPS Example

Historical logs:

```text
U1: old showed A, old propensity = 0.5, click = 1, new model would show A
U2: old showed B, old propensity = 0.2, click = 1, new model would show B
U3: old showed C, old propensity = 0.1, click = 0, new model would show D
U4: old showed A, old propensity = 0.5, click = 0, new model would show A
```

For deterministic new policy:

```text
If new model chooses the same action as logged action:
    pi_new = 1

If new model chooses a different action:
    pi_new = 0
```

Weights:

```text
U1: 1 / 0.5 = 2
U2: 1 / 0.2 = 5
U3: new model wants D, old showed C, so pi_new = 0
U4: 1 / 0.5 = 2
```

IPS numerator:

```text
2 * 1 + 5 * 1 + 2 * 0 = 7
```

IPS estimate:

```text
7 / N
```

This is the basic idea.

### IPS Problem: High Variance

If propensity is very small:

```text
p_old = 0.001
```

then:

```text
weight = 1 / 0.001 = 1000
```

One sample can dominate the estimate.

So IPS can be unbiased under assumptions, but high variance.

---

## 7. SNIPS — Self-Normalized IPS

To reduce variance, we often use SNIPS:

```text
V_SNIPS = sum_i(w_i * r_i) / sum_i(w_i)
```

This normalizes by the total importance weight.

Using the previous example:

```text
weighted reward sum = 2 * 1 + 5 * 1 + 2 * 0 = 7
weight sum = 2 + 5 + 2 = 9

SNIPS = 7 / 9
```

SNIPS is usually more stable than IPS, but it may introduce some bias.

In practice, SNIPS is commonly used as a more robust offline signal.

---

## 8. Doubly Robust Estimation

Doubly Robust combines:

```text
1. A reward model
2. IPS correction
```

The reward model predicts:

```text
expected reward if action a is shown under context x
```

For example:

```text
r_hat(x, a) = predicted click probability
r_hat(x, a) = predicted purchase probability
r_hat(x, a) = predicted watch time
```

Doubly Robust first uses the reward model to estimate outcomes for the new policy, then corrects the prediction using observed logged rewards.

A simplified intuition:

```text
DR estimate =
model prediction for new policy
+ propensity-weighted correction from observed reward
```

A simplified formula:

```text
DR = r_hat(x, pi_new(x))
     + [pi_new(a | x) / pi_old(a | x)] * [r - r_hat(x, a)]
```

Interpretation:

```text
Start with predicted reward under the new policy.
Then use real logged feedback to correct model error where possible.
```

Why it is called doubly robust:

```text
If either the propensity model is correct or the reward model is correct,
the estimator can still be consistent under certain assumptions.
```

In practice, Doubly Robust is often more stable and useful than pure IPS, especially when the action space is large.

## 9. Counterfactual Evaluation for Ranking and Slate Systems

For simple item recommendation, the action is one item.

But in many systems, the action is a ranked list or slate.

Examples:

```text
Top 10 feed items
PDP recommendation rows
Search ranking list
Trending livestream list
Email recommendation slots
```

For slate ranking, the action may be:

```text
[A, B, C, D]
```

or for PDP rows:

```text
[
  "Frequently Bought Together",
  "Compare Similar Items",
  "Sponsored Items"
]
```

This is harder because the number of possible slates is very large.

### Option 1: Position-Level IPS

Treat each position separately:

```text
position 1 action
position 2 action
position 3 action
```

Estimate reward for each position separately.

Pros:

```text
simpler
lower variance
easier to log propensities
```

Cons:

```text
ignores interaction between positions
ignores slate-level diversity and duplication
```

### Option 2: Slate-Level IPS

Treat the entire slate as one action.

Pros:

```text
more theoretically aligned with slate ranking
captures full action
```

Cons:

```text
very high variance
exact slate match is rare
large action space
requires slate-level propensity
```

### Option 3: Doubly Robust Slate Reward Model

Train a model to predict slate-level reward, then correct using randomized logs.

This is often more practical.

### Option 4: Randomized Row Swaps

For row-ranking systems, controlled random row swaps are very useful.

Examples:

```text
randomly swap row 1 and row 2
randomize the order of top 3 eligible rows
randomly include or exclude one module
randomize position for a safe subset of rows
```

This helps estimate:

```text
position effect
row value
interaction effect
exposure bias
```

## 10. Stage 3 — Shadow Testing

After offline and counterfactual evaluation, run the new model in shadow mode.

Shadow testing means:

```text
The new model receives real online requests.
The new model scores or ranks candidates.
But its output is not shown to users.
The old model still serves production traffic.
```

Shadow testing answers:

```text
Can the model run in production?
Are online features available?
Is latency acceptable?
Is the score distribution reasonable?
Are top-ranked items or rows sane?
Does the model have coverage issues?
Does the system fallback too often?
```

Shadow testing can catch:

```text
feature missing bugs
training-serving skew
schema mismatch
latency problems
model loading failures
bad score distribution
unexpected ranking behavior
coverage collapse
duplicate recommendation issues
```

But shadow testing cannot prove user impact because users do not see the new recommendations.

---

## 11. Stage 4 — Canary Rollout

Canary means exposing the new model to a small amount of real traffic.

Example:

```text
1% traffic sees new model
99% traffic sees old model
```

Canary is useful for:

```text
catching severe online product issues
checking real user behavior at small scale
monitoring latency and error rate
ensuring no major metric drop
reducing launch risk
```

Canary is different from shadow testing:

```text
Shadow:
new model runs but users do not see output

Canary:
small user group actually sees new model output
```

Canary may not have enough statistical power to prove lift. It is mainly a safety step before a full A/B test.

---

## 12. Stage 5 — Online A/B Testing

A/B testing is the main online causal validation.

It answers:

```text
Does the new recommendation policy causally improve real user and business metrics?
```

Control:

```text
old recommendation model
```

Treatment:

```text
new recommendation model
```

Randomization unit can be:

```text
user-level
session-level
request-level
geo-level
cluster-level
```

For recommendation systems, user-level randomization is common, but if there are network effects or marketplace interference, cluster-level or geo-level randomization may be needed.

### Primary Metrics

Depending on the product:

```text
CTR
add-to-cart rate
conversion rate
GMV per session
profit per session
watch time
long-watch rate
retention
revenue per user
LTV
```

### Guardrail Metrics

Important guardrails:

```text
latency
error rate
bounce rate
negative feedback
hide / dislike / report rate
diversity
freshness
seller or creator concentration
organic conversion
ad cannibalization
user retention
```

A/B testing is more reliable than offline evaluation because it observes actual user behavior under the new policy.

But A/B tests still have risks:

```text
novelty effects
seasonality
underpowered experiments
network interference
long-term effects not visible in short tests
metric tradeoffs
delayed conversion
```

So A/B testing is the strongest validation stage, but still needs careful design.

## 13. Example: Evaluating a New PDP Row Ranking Model

Suppose we have an e-commerce PDP with multiple recommendation rows:

```text
Compare Similar Items
4 Stars and Above
Frequently Bought Together
Sponsored Items
Customers Also Viewed
```

The new model changes how rows are selected and ordered.

### Offline Evaluation

Use historical PDP logs:

```text
context = user + PDP product + session
action = row shown at each position
reward = row click / add-to-cart / purchase / GMV
```

Evaluate:

```text
AUC for row click prediction
NDCG for row ranking
calibration of row expected value
segment performance by category and device
```

But offline logs are biased because old model determined which rows appeared and where.

### Counterfactual Evaluation

If we have randomized row swaps or propensity logs:

```text
Use IPS / SNIPS / DR to estimate new row policy value.
```

Need logged data:

```text
which row was shown
which position
probability of that row-position assignment
reward
```

For slates, full-slate IPS may be high variance, so position-level IPS or doubly robust slate models may be more practical.

### Shadow Testing

Run new row ranker online but keep old row order visible.

Check:

```text
feature availability
latency
score distribution
row coverage
whether sponsored rows dominate
whether duplicate rows/items increase
fallback rate
```

### Canary

Expose new row ranker to 1% traffic.

Watch:

```text
PDP conversion
row CTR
latency
bounce rate
organic conversion
sponsored revenue
duplicate exposure
```

### A/B Test

Run full experiment.

Primary metric:

```text
profit per PDP view
GMV per PDP view
add-to-cart rate
conversion rate
```

Guardrails:

```text
latency
bounce rate
organic conversion
ad cannibalization
row diversity
user experience
```

## 14. Practical Launch Checklist

Before launching a new recommendation model:

### Offline Checks

```text
AUC / log loss improved
NDCG@K improved
Recall@K improved
Calibration acceptable
Segment performance not worse
No major degradation in important user/product segments
```

### Data Checks

```text
No data leakage
Point-in-time correctness
Training-serving consistency
Feature freshness
Feature missing rate
Candidate coverage
```

### Counterfactual Checks

```text
Propensity logs available
Exploration traffic available
IPS / SNIPS / DR estimate positive or at least safe
Variance not too high
Confidence interval reasonable
```

### Shadow Checks

```text
Online inference works
Latency within SLA
Score distribution reasonable
Top recommendations sane
No coverage collapse
Fallback rate acceptable
```

### Canary Checks

```text
No major drop in business metrics
No latency issue
No large increase in negative feedback
No major segment-level issue
```

### A/B Test Checks

```text
Primary metric improves or is neutral with strategic benefit
Guardrails pass
Segment results make sense
No unexpected ecosystem harm
Decision is consistent with pre-defined launch criteria
```

## 15. Common Interview Framing

A strong answer:

```text
For a recommendation model architecture change, I would evaluate it through a multi-stage process. First, I would run offline evaluation using time-based historical validation and metrics such as AUC, log loss, Recall@K, NDCG@K, and calibration. This helps filter out bad models, but it is not sufficient because recommendation logs are generated by the old policy.

The main issue is exposure bias: users only provide feedback on items or rows they actually saw. For actions the old model never exposed, we do not observe counterfactual rewards. If we have randomized exploration and propensity logging, I would use counterfactual evaluation methods such as IPS, self-normalized IPS, or doubly robust estimation.

After that, I would run the model in shadow mode to check feature pipelines, latency, score distribution, and ranking sanity. Then I would use a small canary rollout to catch severe production issues. Finally, I would run an online A/B test to measure the causal impact on business metrics such as CTR, CVR, GMV, retention, or long-term engagement, while monitoring guardrails such as latency, diversity, safety, and user experience.
```

---

## 16. One-Line Summary

```text
Offline backtesting tells us whether the new model looks better on historical data.
Counterfactual evaluation estimates how it might have performed under old logs.
Shadow testing checks production readiness.
Canary reduces launch risk.
A/B testing proves real online causal impact.
```

The core difference from standard supervised learning is:

```text
In recommendation systems, labels are policy-dependent.
The model decides what users see, and what users see determines what labels we observe.
```
