Approach A — Nebius Token Factory API

Uses the OpenAI-compatible REST API. No GPU provisioning needed — just upload your data, submit a job, and download the checkpoint.

Prerequisites

  • A Nebius account with Token Factory access
  • An API key exported as NEBIUS_API_KEY
  • Training data in JSONL format (see Dataset Format)
# Install the OpenAI SDK (Token Factory uses the same API surface)
pip install openai
bash

Step 1 — Initialize client

import os
from openai import OpenAI

client = OpenAI(
    base_url="https://api.tokenfactory.nebius.com/v1/",
    api_key=os.environ["NEBIUS_API_KEY"],
    # ^ Set via: export NEBIUS_API_KEY="your-key-here"
)
python

Step 2 — Upload datasets

# Upload training file
train_file = client.files.create(
    file=open("training.jsonl", "rb"),
    purpose="fine-tune",
)
print(f"Training file ID: {train_file.id}")

# Upload optional validation file (recommended: 10–20% of data)
val_file = client.files.create(
    file=open("validation.jsonl", "rb"),
    purpose="fine-tune",
)
print(f"Validation file ID: {val_file.id}")
python

Step 3 — Create fine-tuning job

job_request = {
    "model": "meta-llama/Llama-3.1-8B-Instruct",
    # ^ Base model to fine-tune. Must be on Nebius's supported list.

    "training_file": train_file.id,
    "validation_file": val_file.id,  # optional but recommended

    "suffix": "math-aimo",
    # ^ Appended to the model name for identification (e.g. llama-3.1-8b-math-aimo)

    "seed": 42,
    # ^ Random seed for reproducibility across runs

    "hyperparameters": {
        "n_epochs": 3,
        # ^ Full passes over the dataset. More → specialization, less → generality.

        "batch_size": 8,
        # ^ Samples per gradient step. Range: 8–32. Larger = more stable but more VRAM.

        "learning_rate": 1e-5,
        # ^ Step size. Range: 1e-6 to 5e-5. Too high → diverge; too low → slow.

        "context_length": 4096,
        # ^ Max tokens (prompt + completion). Set to cover P95–P99 of your examples.
        # ^ Longer = more VRAM. Analyze your data distribution first.

        "warmup_ratio": 0.05,
        # ^ Fraction of steps for LR warm-up (linear ramp from 0 → peak).

        "weight_decay": 0.01,
        # ^ L2 regularization. Helps prevent overfitting on small datasets.

        "lora": True,
        # ^ Enable LoRA (Low-Rank Adaptation). Trains only small adapter matrices.
        # ^ Much less VRAM than full fine-tune. Set False for full FT.

        "lora_r": 16,
        # ^ LoRA rank. Higher = more expressive but more memory. Range: 8–128.

        "lora_alpha": 32,
        # ^ Scaling factor. Effective LR multiplier = lora_alpha / lora_r.
        # ^ Set to 2× lora_r as a baseline.

        "lora_dropout": 0.05,
        # ^ Dropout on LoRA layers. 0 = no dropout. 0.05–0.1 adds regularization.

        "packing": True,
        # ^ Pack multiple short examples into one sequence up to context_length.
        # ^ Maximizes GPU utilization. Disable only if example ordering matters.

        "max_grad_norm": 1.0,
        # ^ Gradient clipping threshold. Prevents exploding gradients.
    },

    # Optional: push checkpoint to HuggingFace Hub after training
    "integrations": [
        {
            "type": "wandb",
            "wandb": {
                "project": "aimo-finetune",
                "name": "llama-3.1-8b-math-run1",
                "entity": "your-wandb-username",
            }
        },
        # Hugging Face push (optional)
        # {
        #     "type": "huggingface",
        #     "huggingface": {
        #         "output_repo_name": "your-hf-username/my-math-model",
        #         "api_token": os.environ["HF_TOKEN"],
        #     }
        # }
    ],
}

job = client.fine_tuning.jobs.create(**job_request)
print(f"Job created: {job.id}  status: {job.status}")
python

Step 4 — Poll until complete

import time

TERMINAL = {"succeeded", "failed", "cancelled"}

while job.status not in TERMINAL:
    time.sleep(15)  # poll every 15 seconds — avoid hammering the API
    job = client.fine_tuning.jobs.retrieve(job.id)
    print(f"Status: {job.status}  |  Trained tokens: {job.trained_tokens}")

if job.status == "succeeded":
    print("Training complete!")
else:
    print(f"Job failed: {job.error}")
python

Step 5 — Download checkpoints

import pathlib

checkpoints = client.fine_tuning.jobs.checkpoints.list(job.id)

for ckpt in checkpoints.data:
    print(f"Checkpoint {ckpt.id}  step {ckpt.step_number}")

# Download the latest checkpoint's files
latest = checkpoints.data[-1]

for file_id in latest.result_files:
    content = client.files.content(file_id)
    out_path = pathlib.Path(f"checkpoint_{latest.step_number}_{file_id}")
    out_path.write_bytes(content.content)
    print(f"Saved: {out_path}")
python
Tip
Use the latest checkpoint for deployment unless you have a reason to use an earlier one (e.g. validation loss spiked at the end).

Approach B — Self-Hosted GPU VM (unsloth + TRL)

Provision a Nebius GPU VM, run setup.sh, and train with your own loop. Gives full control over optimizer, LoRA config, and training code.

  1. Create a VM on Nebius Console

    Go to Compute → Create Instance. Recommended spec:

    FieldRecommended
    ImageUbuntu 22.04 + CUDA 12.x
    GPUH100 SXM5 80GB × 1–4
    RAM128–256 GB
    Disk500 GB SSD
    SSH keyPaste ~/.ssh/id_nebius.pub
  2. Connect via SSH 
    ssh -i ~/.ssh/id_nebius ubuntu@<nebius-ip>
    bash

    Add to ~/.ssh/config for convenience:

    Host nebius
    HostName <nebius-ip>
    User ubuntu
    IdentityFile ~/.ssh/id_nebius
    ssh config
  3. Run the one-shot setup script 
    # From your Mac — upload the script
scp "Chapter LLM Nebius Setup/setup.sh" nebius:~/setup.sh

# On the server — install everything (~10–15 min)
bash ~/setup.sh
    bash

    Installs: PyTorch (CUDA-matched) → unsloth → trl → peft → vLLM → wandb → clones your repo.

  4. Start training inside tmux 
    tmux new -s train

source ~/llm_env/bin/activate
cd ~/Journey_of_MachineLearning/"Chapter LLM SFT"
python sft_example.py 2>&1 | tee training.log

# Detach without killing: Ctrl+b, then d
# Reattach later: tmux attach -t train
    bash
  5. Monitor GPU usage 
    watch -n 2 nvidia-smi     # GPU util should be 90–100%
nvtop                     # nicer htop-style GPU view
tail -f training.log      # live loss output
    bash
  6. Multi-GPU: use torchrun 
    export CUDA_VISIBLE_DEVICES=0,1
torchrun --nproc_per_node=2 sft_example.py
    bash
  7. Stop the VM when done — GPU time is billed per minute 
    sudo shutdown now
    bash
Cost warning
H100 80GB instances cost ~$2–4/hour. Always stop the VM when not training. Use tmux so a dropped SSH connection doesn't kill your job.

Hyperparameter Reference

Applies to both approaches (Token Factory fields map 1:1 to TRL/unsloth args).

Parameter Range Default Notes
n_epochs 1–20 3 More epochs → specialization; risk of overfitting
batch_size 8–32 8 Larger = more stable gradients, more VRAM
learning_rate 1e-6 – 5e-5 1e-5 Too high → diverge; too low → slow. Try 2e-5 first
context_length int (tokens) 8192 Cover P95–P99 of your examples. Longer = more VRAM
warmup_ratio 0–1 0 Fraction of steps for LR warm-up. 0.05 recommended
weight_decay ≥ 0 0 L2 regularization. 0.01–0.1 for small datasets
lora bool false Enable LoRA adapters. Recommended for 7B+ models
lora_r 8–128 8 LoRA rank. Higher = more expressive, more VRAM
lora_alpha ≥ 8 8 Scaling factor. Typically set to 2× lora_r
lora_dropout 0–1 0 Dropout on LoRA layers. 0.05 adds light regularization
packing bool true Pack short examples into full-length sequences. Boosts GPU util
max_grad_norm ≥ 0 1 Gradient clipping. 1.0 is a safe default
MUON optimizer (self-hosted only)
When using Approach B, you can replace AdamW on LoRA B-matrices with MUON for faster convergence. See sft_lora_muon.py for the implementation. MUON is not available via the Token Factory API.

Dataset Format (JSONL)

Both approaches expect JSONL (one JSON object per line), in the OpenAI conversation format: 

// training.jsonl — each line is one training example
{"messages": [
  {"role": "system",    "content": "You are a math problem solver..."},
  {"role": "user",      "content": "What is the sum of 1 to 100?"},
  {"role": "assistant", "content": "Using n(n+1)/2: 5050.\n\\boxed{5050}"}
]}
{"messages": [
  {"role": "system",    "content": "You are a math problem solver..."},
  {"role": "user",      "content": "How many primes are less than 20?"},
  {"role": "assistant", "content": "2,3,5,7,11,13,17,19 → 8 primes.\n\\boxed{8}"}
]}
jsonl

Generate JSONL from Python dicts

import json

data = [
    {"problem": "...", "solution": "..."},
    # ... more examples
]

SYSTEM = "You are a math problem solver. Show reasoning, answer in \\boxed{}."

with open("training.jsonl", "w") as f:
    for row in data:
        record = {"messages": [
            {"role": "system",    "content": SYSTEM},
            {"role": "user",      "content": row["problem"]},
            {"role": "assistant", "content": row["solution"]},
        ]}
        f.write(json.dumps(record) + "\n")
python
Dataset size tips
Split 80–90% train / 10–20% validation. Analyze token length distribution and set context_length to cover P95–P99 of your examples — anything shorter gets truncated and anything longer adds padding overhead.

Transient errors

5xx errors
For transient server errors (5xx), safely resubmit the same job request. The API is idempotent within a session if you use the same seed.
